## Status

🟡 In Progress

## Owner

Engineering

## Priority

## Objective

Expose secure, paginated, filterable, sortable, and searchable REST APIs for all CRM entities, including bulk operations.

## Requirements

- [x] CRUD APIs for all CRM entities
  - leads
  - companies
  - contacts
  - deals
  - tasks
  - notes
  - activities
- [x] Pagination on list endpoints
- [x] Filtering on list endpoints
- [x] Sorting on list endpoints
- [x] Search on list endpoints
- [x] Bulk operations (create, update, delete)

## Acceptance Criteria

- Every entity supports `GET`, `POST`, `PATCH`, and `DELETE` via standard REST routes.
- List endpoints accept `page`, `pageSize`, `search`, `sortBy`, `sortDirection`, and filter query params.
- Bulk endpoints accept arrays or `ids` payloads and enforce organization scoping.
- All endpoints authenticate users and authorize actions via RBAC.
- Inputs are validated with Zod schemas.
- Responses follow the standardized API contract.

## Notes

- CRM services were refactored to accept an optional Supabase client so that API routes can inject the server client while browser usage remains backward compatible.
- Organization scoping is enforced at the API layer by injecting `organization_id` into filters and payloads.
- A generic entity route factory (`app/api/crm/lib/factory.ts`) reduces duplication across the seven CRM resources.
